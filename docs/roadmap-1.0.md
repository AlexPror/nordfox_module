# Дорожная карта до релиза 1.0

Краткий план доработок NordFox Module Manager до зрелого релиза. Документ можно обновлять по мере реализации.

## Оценка текущей зрелости

- Рабочий **beta / early production** для узкого инженерного сценария.
- Уже есть: изоляция COM (subprocess для сборки), локальный сервис PDF/DWG, fallback-экспорт, объединение PDF, диагностика в логах.

## Чеклист «готово к 1.0»

### 1) Стабильность ядра

- Матрица совместимости: версия КОМПАС × Windows × Python.
- Единые коды ошибок (`E_*`) и короткие подсказки в UI.
- Ограниченный retry для безопасных операций.
- Мягкий reconnect к COM/сервису без полного перезапуска UI.

### 2) Окружение при старте

- Preflight: доступность COM API5/API7, права на папки, наличие `pypdf`, путь к Python.
- Экран «Диагностика» и кнопка «Скопировать отчёт».

### 3) Регрессия

- 5–10 эталонных проектов (малый/средний/сложный).
- Сценарии: сканирование, переменные, обозначения/наименования, штампы, PDF (+ merge), DWG.
- Критерий: нет падения UI; допустимы только классифицированные предупреждения.

### 4) Наблюдаемость и UX

- Итог операции: успех / частичный успех / провал.
- Журнал с привязкой к операции (по возможности).
- Идея на финиш: «Открыть папку результата» после экспорта.

### 5) Локальный сервис

- Поддерживать актуальный `/health` и версию backend.
- Явный статус сервиса в UI; избегать параллельных конфликтующих COM-операций.

### 6) Документация и релиз

- В README: поддерживаемые версии КОМПАС, ограничения, troubleshooting.
- Changelog и release checklist.

### 7) Приёмка версии 1.0

- 2–3 недели без критических падений в реальной работе.
- Не менее 95% успешных прогонов smoke-набора.
- Нет «тихих» сбоев: всё отражено в UI и логах.
- Воспроизводимая установка на чистой машине.

## Спринт 1 — стабильность и предсказуемость

**Цель:** предсказуемое поведение и понятные ошибки без «рандома» COM.

| Приоритет | Задача |
|-----------|--------|
| P0 | Preflight при старте (сначала предупреждение, не жёсткий stop без явной настройки). |
| P0 | Единые коды ошибок + текст подсказки в UI. |
| P0 | Блокировка конкурирующих тяжёлых операций (очередь или понятный статус «операция выполняется»). |
| P1 | Единый summary результата (успех / частичный / ошибка). |
| P1 | Структурированные логи + отчёт для поддержки. |

**Критерий готовности:** smoke без падений UI; любая ошибка понятна пользователю; нет гонок тяжёлых задач.

## Спринт 2 — регрессии и production-ready

**Цель:** уверенность в релизе и удобство ежедневной работы.

| Приоритет | Задача |
|-----------|--------|
| P0 | Расширить smoke-набор, зафиксировать baseline. |
| P0 | Таблица совместимости в README. |
| P1 | Watchdog локального сервиса (перезапуск только при «мёртвом» health + backoff). |
| P1 | UX: открыть папку вывода, повтор только неуспешных файлов. |
| P1 | Процесс релиза: `1.0.0-rc1`, changelog, checklist. |

**Критерий готовности:** smoke ≥95%; стабильность на бою; готов RC к `1.0.0`.

## Риски для существующего функционала

- **Низкий риск:** диагностика, коды ошибок, логи, документация.
- **Средний риск:** слишком жёсткая блокировка UI или агрессивный перезапуск сервиса — смягчать флагами, backoff и регрессией.
- Рекомендация: минимальные диффы, отдельные коммиты, прогон smoke перед merge.
